# 研判分析思路

在攻防演习期间对告警事件进行研判分析时，往往需要考虑以下三个关键问题：

1.告警事件是否为误报？

2.攻击事件是否为攻防演习中攻击方发起？

3.攻击是否成功？

## 是否为误报

对于一个告警事件的研判，首先需要判断该告警事件是否为误报。这里的误报主要指实际没有遭受到攻击事件，但是由于各种原因导致安全产品产生了错误的告警信息。 目前常见的导致安全产品产生误报的情况包含：

1.对于一些攻击采用了较为宽泛的规则匹配，容易产生误报。

2.使用了智能算法引擎识别攻击，但是算法实现不好，容易产生误报。

3.一些特殊的应用场景，流量刚好匹配到了检测规则，但实则是正常的业务流量。这一类误报场景比较多见的如：业务正常的响应内容为十六进制的字符串触发了冰蝎 WebShell 的检测规则。另外，需要注意的是在一些比较老旧的系统，会存在参数直接拼接 SQL 语句进行查询，这些系统本身存在 SQL 注入漏洞，因此正常的业务流量会触发 SQL 注入告警，对于这种情况，如果需要研判是否真实遭受了 SQL 注入攻击则需要分析流量中的SQL 语句是否存在恶意操作。

判断一个告警是否为误报，需要对告警的流量进行分析，比如查看告警详情，查看请求头、请求体内容，主要分析流量中可能是哪些特征触发了告警，确认流量是否包含攻击载荷，如果不包含攻击载荷，则为误报。

另外，对于误报的判断还可以使用排除的方法，例如对于攻防演习期间产生的一个告警事件，如果难以判断是否为误报事件，可以根据历史告警事件来判断该事件是否为误报或者 是否与攻防演习相关的事件，如果在攻防演习开始前就一直存在该事件的告警，则大概率会是误报事件或者与攻防演习无关的事件。

## 是否为攻防演习攻击事件

判断攻击事件是否与攻防演习相关，可以结合以下两个关键要素进行研判。

1\. 攻击事件类型

2\. 攻击 IP

### 攻击事件类型

在攻防演习中，攻击方所发起的攻击主要以获取权限、数据为目标，我们可以从攻击事件对目标造成的影响出发，总结历年攻防演习中攻击者使用的高频攻击，对攻击事件进行分类，从而判断攻击是否为攻防演习中的攻击方发起。

针对 WEB 应用的攻击

1.文件上传

文件上传漏洞是指应用程序在实现文件上传功能时，没有对用户上传的文件类型做严格校验，导致用户可上传任意类型的文件，攻击者可上传 WebShell对服务器进行控制。

常见的文件上传包括应用自实现文件上传、编辑器、中间件部署等。

应用自实现上传：主要指应用开发自实现的文件上传功能， 由于代码对上传文件的类型 没有进行严格的校验或者校验可被绕过导致攻击者可上传 WebShell 。如泛微 OA 、致远 OA 等任意文件上传漏洞。

第三方编辑器：应用常常会使用第三方的编辑器，一些编辑器历史版本曾被披露过文件 上传漏洞，如 FCKeditor 、eWebEditor 、UEditor 、CKFinder 等。

中间件部署：中间件控制台实现了部署应用的功能，攻击者通过默认口令、弱口令等登录进去控制台部署WebShell。

2.命令执行

命令执行漏洞是指应用没有对用户的输入做过滤或者校验等，用户的输入可直接传入可以执行操作系统命令的函数中，导致可执行任意命令。命令执行漏洞常见于运维或者安全产品，因为这些产品有些功能往往需要调用操作系统命令实现，在历年攻防演习中常见的命令执行漏洞包括齐治堡垒机、深信服 VPN 、深信服 EDR 等远程命令执行漏洞。

3.代码执行

代码执行漏洞是指应用没有对用户的输入做过滤或者校验等，导致攻击者可直接传入任意恶意代码进行执行。常见的远程代码执行漏洞有 Struts2 远程代码执行、ThinkPHP 远程代码执行、Jeekins 远程代码执行、泛微 OA beanShell 组件远程代码执行等。

4.反序列化

反序列化漏洞是指应用对用户可控制的数据进行反序列化时，攻击者能够操作序列化的对象，以将有害的数据传递到应用代码中。攻击者利用反序列化漏洞可实现远程代码执行， 进而获取服务器权限。反序列化漏洞常见于 JAVA 语言，如 WebLogic 、Jboss 、WebSphere 等中间件都曾被披露过 JAVA 反序列化漏洞。此外，Shiro 反序列化漏洞和 Fastjson 反序列化 漏洞攻击也是最近两年的攻击中广受攻击方青睐的漏洞。除了JAVA 之外，.NET 语言也同样会出现反序列化的漏洞，如 Exchange 反序列化漏洞。

5.WebShell

WebShell 是指攻击者在进行 WEB 应用攻击时上传到服务器进行控制的代码文件，其本 质是利用代码执行，可对服务器进行文件管理、命令执行等操作。攻击者在利用文件上传、 命令执行、代码执行、反序列化等漏洞时，最终往往会将 WebShell 写入 WEB 应用目录以方便维持对服务器的控制。

6.SQL 注入

SQL 注入攻击是指 WEB 应用程序对用户输入的数据的合法性没有校验或者过滤不严， 攻击者输入的恶意 SQL 语句拼接在 WEB 应用原来定义的查询语句，从而改变了原有的 SQL 查询语句逻辑。利用 SQL 注入，攻击者可获取数据库中的数据信息。另外，在一些特定的数 据库中，利用SQL 注入也可获取服务器权限。如果后端数据库为 MSSQL ，攻击者可能利用 SQL 注入恢复 xp\_cmdShell 存储过程等直接执行命令获取服务器权限；如果后端数据库为  MySQL ，攻击者可能利用 SQL 注入通过 load\_fille 函数读取服务器文件或者通过 into outfile 写入 WebShell。

针对 Windows 的攻击

攻防演习中，针对 Windows 的攻击主要包括口令、域、MS17-010 、CVE-2019-0708 等。

1\. 口令

针对 Windows 的口令攻击主要包括：弱口令暴力破解、口令获取及复用。攻击者在进行 Windows 登录时，一般会使用 SMB 或者 RDP 这两种登录方式。

2.域

3.MS17-010

MS17-010  (永恒之蓝) 攻击也是历年攻防演练中攻击方常用的攻击 Windows 操作系统 的漏洞攻击之一。该漏洞是利用了 Windows 服务器消息块 (SMB) v1 的漏洞，在未安装相 关补丁且 445 端口未关闭的情况下，攻击者可利用该漏洞攻击获取机器权限。

4.CVE-2019-0708

CVE-2019-0708 是 Windows 远程桌面服务的远程代码执行漏洞，该漏洞无需用户交互。 在未安装相关补丁且远程桌面端口 (常见为 3389 端口) 开放的情况下，攻击者可利用该漏洞攻击获取机器权限。需要注意的是，该漏洞攻击容易造成系统蓝屏。

针对 Linux 的攻击

相比 Windows，针对 Linux 的攻击可直接获取权限的漏洞相对比较少，主要还是以 SSH 口令为主，常见的攻击如 SSH 弱口令暴力破解、SSH 口令获取及复用。

其他

对于利益驱动相关的告警事件，如利用漏洞下载执行挖矿病毒、勒索病毒、僵尸蠕虫等， 或者造成实际危害性质的如拒绝服务攻击等，一般都是非攻防演习相关事件。

### 攻击 IP

对于符合以上攻防演习中高频的攻击事件类型，可进一步结合攻击 IP 研判是否为攻防演习相关的攻击行为。这里的攻击 IP 包括攻击源、C2 、VPS 等 IP 信息。

可通过互联网威胁情报、攻防演习中台输出的黑 IP 情报等方式确认攻击 IP 是否在攻防 演习 IP 名单内。或者通过互联网搜索等方式确认攻击 IP 是否为被披露的挖矿病毒、勒索病毒、 僵尸蠕虫攻击事件所属 IOC ，如果符合则表明告警事件大概率为非攻防演习攻击事件。

## 攻击是否成功

研判攻击是否成功，可以分两种攻击类型去讨论：针对 WEB 应用的攻击和非 WEB 的其他攻击。

对于 WEB 应用的攻击研判，研判攻击是否成功的关键要素为响应状态码、请求报文和 响应报文。研判攻击成功与否，可以先通过响应状态码做初步简单判断，通过响应状态码可以快速识别部分攻击成功与否，如果通过响应状态码不能判断的话，则下一步可以结合请求 与响应进行研判。

对于非 WEB 攻击，如果直接从流量的内容进行研判的话，大多漏洞攻击的流量会比较难研判。

### 响应状态码

**404**：404 状态码表示请求资源不存在，即表示攻击失败。如下图，攻击者通过批量扫描常见的 WebShell 以发现目标系统上历史被植入的 WebShell，从而触发了“WebShell 后门程序访问控制” ，响应状态码为 404 可知攻击不成功。

![](image/image_JC5QtffbbJ.png)

不管是对于什么WEB 漏洞攻击的研判，响应状态码都是研判成功与否的首要研判依据， 如果响应状态码为 404 基本可以研判攻击失败，也就无需再根据请求响应等进一步研判了。 	(当然，这并不是绝对的，也有例外的情况，攻击者在一些情况下也可以篡改响应状态码，  如 WebShell 的响应状态码。现在这种情况不多见，暂时可以先不考虑)  (备注：在指导手册 第三部分的高频漏洞研判，默认都是请求状态为非 404 状态前提下的研判，不再重复响应码 404 状态码的研判思路)

**200**：200 状态码表示请求成功，但是请求成功并不代表攻击成功，具体需要结合请求与 响应进行判断。如下图攻击中，攻击者尝试利用 Struts2 远程代码执行漏洞 (S2-045) 对目标 系统进行攻击，响应状态码为 200 。该漏洞攻击载荷在 HTTP 请求头部的 Content-Type ，分 析该攻击载荷，如果漏洞存在的话会在响应的头部添加testvuln 字段，值为 1234x 1234 即  1522756 。分析响应的头部，并未发现存在testvuln 字段，因此研判该漏洞攻击未成功。

![](image/image_1_LNMFM8jQUG.png)

**401**：401 状态表示未授权状态。该状态码返回常见于 HTTP 的 Basic 认证，如 Tomcat 的控制台就是使用 Basic 认证，如果未提交授权认证信息或者授权认证失败。Basic 认证请求 最明显的特征为请求的头部存在 Authorization: Basic Basic64 字符串。其中 Base64 字符串 为“用户名:口令”的 base64 编码，解码后即可获取用户名及口令明文。在攻防演习中，攻 击者可能会通过暴力破解、默认口令尝试等获取 Basic 认证的凭据。如果在某个时间段内出现 大量响应状态码为 401 的请求，则大概率是攻击方在对 Basic 认证进行暴力破解，如果 Basic 认证成功则响应状态码为 200。因此，依靠响应状态码可以研判对于 Basic 认证的攻击是否成功。

如下图响应状态码为 401 ，为 Basic 认证失败的请求：

![](image/image_2_SaE53n9AGI.png)

响应状态码为 200 ，表明 Basic 认证成功。对认证信息进行 base64 解码得知 Tomcat 控制台的用户名及口令为 tomcat:tomcat。

![](image/image_3_yoyg_aTUrF.png)

**500**：500 状态码表示服务器内部错误，通常漏洞攻击也会导致出现 500 错误，但是出 现 500 错误并不表示攻击失败，需要根据实际情况研判。如下图，Fastjson反序列化漏洞攻 击，响应状态为 500 ，但是该攻击是成功的，由于漏洞攻击触发了 DNS 查询，因此在 Burp Collaborator 中出现了 DNS 查询的日志。

![](image/image_4_cpqsxhBRhp.png)

### 请求与响应

研判攻击非常关键的要素在于流量上下文，即请求和响应内容。通过分析攻击请求，可 以清晰了解攻击者的攻击意图，如利用漏洞攻击执行了什么命令等。进一步的分析响应内容， 判断服务器是否对攻击的请求进行了一致的响应，如果响应和攻击请求一致则表明了攻击成功，漏洞存在。

因此这里根据请求与响应进行研判比较适合于针对 WEB 应用的攻击。

对于 WEB 攻击，我们可以分漏洞攻击结果是否有回显两种情况进行研判。

如果漏洞攻击结果存在回显，可以直接根据请求和响应内容来研判攻击是否成功。 以  S2-016 远程代码执行漏洞为例，下图中左边红框部分就是请求中 S2-016 漏洞攻击的关键攻击载荷。

![](image/image_5_YZUnJwkEap.png)

上面的载荷已经经过 URL 编码处理了，可以先做 URL 解码。

![](image/image_6_-0nwtavTu-.png)

分析攻击载荷，载荷主要实现通过 java.lang.ProcessBuilder 执行 whoami 命令，通过  context.get('com.opensymphony.xwork2.dispatcher.HttpServletResponse')获取 HTTP 的响应，并且将执行命令的结果写入响应中。也就是说，如果以上的攻击成功的话，将会在响应中返回执行 whoami 命令的结果。查看响应内容，发现确实返回了whoami 命令执行的结果， 当前用户为 root ，也就是说经过研判得知漏洞攻击是成功

![](image/image_7_Pdt-Bkirby.png)

如果漏洞攻击结果不回显，研判会相对比较复杂一点，需要结合漏洞和攻击行为依据实 际情况讨论。

以 SQL 注入漏洞攻击为例，SQL 注入的布尔盲注和时间盲注两种注入方式是没有直接 在响应回显注入结果的。如下图，SQL 注入的数据包中，简单分析下请求和响应数据包，可以发现请求中通过 GET 请求的方式提交了 SQL 注入的语句，但是响应包中为正常的页面信息，不能直接观察请求和响应的内容来进行研判。

![](image/image_8_H4scZm62Y3.png)

对 SQL 注入的语句进行 URL 解码后得到：

![](image/image_9_df8wvSIlaK.png)

可以发现以上的SQL 注入的语句有多个嵌套，从最里层进行分析：

SELECT IFNULL(CAST(COUNT(DISTINCT(schema\_name)) AS CHAR),0x20) FROM INFORMATION\_SCHEMA.SCHEMATA：COUNT 为统计数量的函数，schema\_name 表示数据库名，INFORMATION\_SCHEMA.SCHEMATA 可获取所有数据库信息，综上该语句旨在 获取数据库的数量。

IF(ORD(MID((SELECT IFNULL(CAST(COUNT(DISTINCT(schema\_name)) ASCHAR),0x20) FROM INFORMATION\_SCHEMA.SCHEMATA), 1, 1))>48,0,5) ：MID 为取位函 数，MID(xxx, 1, 1)即取 xxx 的第一位，然后 ORD 为计算 ASCII 码函数，IF 为条件判断，IF(条 件,0,5)即如果条件成立则结果为 0，否则为 5，综上该语句旨在判断数据库数量的第一位 ASCII 码值是否大于 48。

root' AND (SELECT \* FROM (SELECT(SLEEP(5-(IF(ORD(MID((SELECTIFNULL(CAST(COUNT(DISTINCT(schema\_name)) AS CHAR),0x20) FROM INFORMATION\_SCHEMA.SCHEMATA), 1, 1))>48,0,5)))))MYuG) AND 'fISA'='fISA：SLEEP 为 MySQL 的时间延迟函数，整个 SQL 注入语句为如果数据库数量的第一位数字 ASCII 码大 于 48 的话则延迟 5 秒，否则不延迟。

对于以上攻击的研判，可以使用数据包重放的方法，重放数据包看是否产生时间延迟， 如果产生则说明攻击成功。但是需要注意的是以上只是时间盲注的大量数据包中的一条，并且我们不确定以上以上的 IF 判断条件是正确的，因此可以将 SQL 语句进行简化，看是否产生时间延迟即可确认漏洞是否存在，如存在则证明以上的漏洞攻击是成功的。

将 SQL 语句修改为 root' AND SLEEP(5) AND 'fISA'='fISA，可以看到响应延迟了5063ms， 即约5 秒。证明 SQL 语句执行了，即存在 SQL 注入漏洞。

![](image/image_10_cq45Nea_WD.png)

进一步的可以将 SQL 语句修改为 root' AND SLEEP(0) AND 'fISA'='fISA 进行二次确认。 如下，SLEEP(0)即不产生时间延迟。

![](image/image_11_gkE3-Rr9dD.png)

当然，对于不同的无回显漏洞攻击场景研判方法也不尽相同，有时不需要进行漏洞重放，也能判断漏洞攻击是否成功。如以上 SQL 注入漏洞攻击中，通过 User-Agent 得知攻击者是使用了 sqlmap 工具进行了 SQL 注入攻击，而 sqlmap 注入工具在漏洞利用前会进行漏洞验证， 如果漏洞验证不存在则退出运行，也就是说在漏洞不存在的情况下只会产生简单的漏洞验证流量，不会产生漏洞利用的流量。而上面的攻击流量为漏洞利用过程中获取数据库信息的流量，也间接证明了漏洞是存在的。

特别需要注意的是，攻防演习期间，进行攻击流量重放需要征求客户方的同意，切勿私自重放流量以被误当作为攻击方的攻击流量。另外，在进行流量重放前，需要修改原攻击性质的载荷为对系统不产生实质危害的验证性载荷。

### 漏洞攻击与应用

对于研判攻击是否成功，使用排除法也可以节省很多不必要的攻击研判。这里的排除法主要是判断当前应用的中间件、开发语言、数据库类型等是否与攻击的漏洞相匹配，例如使用 PHP 开发语言的应用，那么出现 WebLogic 反序列化等 JAVA 语言相关的告警事件则可以直接忽略此类事件。

### 其他

对于非 WEB 攻击的研判分析可以根据攻击手法以及告警置信度两个维度来辅助进行研判。

由于非 WEB 攻击会较多的集中在内网渗透阶段，因此可以把研判的重点放在内网渗透攻击攻击者常用的攻击手法。在内网渗透阶段，攻击者的主要目标有三个：

1.快速、批量获取大量机器权限；

2.获取关键系统权限；

3.突破进入生产等核心网络区域。

攻击者往往会通过口令的获取及复用、MS17-010 以及 CVE-2019-0708 等漏洞、获取 Windows 域管理员权限、 获取运维堡垒机或者 EDR 等集中管控设备的权限等手段来获取内网中大量机器的权限。而攻击者如果想要突破网络隔离等限制，则经常会使用寻找一些双网卡机器、获取路由交换器等网络设备权限等手段。

综上，可以关注以下非 WEB 相关攻击事件。

1.SSH 、Windows 口令登录

2.Telnet 登录

3.MS17-010 、CVE-2019-0708 攻击事件

攻击者在内网横向移动的过程中，常常会对内网进行信息收集， 以及口令的复用。因此我们可以关注内网中是否出现单一机器对一个或多个网段的 主机大量端口发起访问的行为， 以及单一机器对一个或多个网段的主机发起 SSH 、SMB 或者 RDP 的登录请求行为。

另外，我们也可以从告警事件的置信度辅助进行研判。 目前安全产品可以实现对一些攻击事件比较精准研判，一般来说，如果内网主机出现一些高精准的告警事件，大概率上说明内网中可能有主机已经被攻陷了。

如下图中，ISOP 平台先后出现了两个告警事件：MS17-010 漏洞攻击、后门程序  Doublepulsar 通信。一般来说这些告警事件的置信度都比较高，而且 Doublepulsar 也是 MS17-010 攻击中向目标机器植入后门的过程之一，根据这两个关联的置信度高攻击基本可以 研判源 IP 192. 168. 116. 139 已经失陷，并且，该机器被当作跳板向内网主机 192. 168. 116. 174 发起了 MS17-010 漏洞攻击。

# 高频漏洞研判思路

1. WEB

   SQL 注入漏洞

漏洞概述：

所谓SQL 注入，就是通过把 SQL 命令插入到请求的参数中，最终达到欺骗服务器执行 恶意的SQL 命令。具体来说，它是利用现有应用程序，将恶意 SQL 命令注入到后台数据库引擎执行的能力，它可以通过输入恶意 SQL 语句获取数据库敏感信息或者执行恶意操作系统命令等。

研判分析：

对于 SQL 注入漏洞的研判，存在多种研判思路，在不同的情况下可以考虑使用不同的方法进行分析。

1.过程分析法

因为利用SQL 注入漏洞获取数据库的数据往往需要一个比较固定的利用过程，比如从漏洞的探测再到漏洞的利用。而在漏洞的利用过程一般又会经历获取数据库名—>获取表名—>获取列名—>获取具体数据。因此，我们可以通过分析 SQL 注入的流量是否包含了完整的攻击过程来进行研判，如果只出现了漏洞的探测过程流量，而没有深入的漏洞利用过程，那就大概率表明漏洞不存在，攻击不成功。过程分析法比较适合用于一些非通用应用的 SQL 注入 漏洞攻击上，因为通用应用的数据库结构一般一致，如具有固定的表名和列名，攻击方可直接通过已获知的表名和列名情况，直接构造 SQL 漏洞利用语句直接获取数据。

如下图为使用sqlmap 工具进行 SQL 注入测试的流量，从流量可以看到基本都是简单的探测是否存在 SQL 注入漏洞，而没有后续漏洞利用的过程，因此大概率不存在 SQL 注入漏洞。

![](image/image_13__hVnLBoA6g.png)

而在下图的流量中存在获取数据库名的流量特征，对于这种情况则大概率是攻击成功的。

![](image/image_14_UnNejDAkGy.png)

总结常见数据库在获取数据库敏感信息 (这里暂不考虑命令执行等利用方式) 的漏洞利用过程中的关键特征，如果出现比较连续有序的漏洞利用过程，且出现了这些特征，需要引起注意。

MySQL：

INFORMATION  SCHEMA 、schema  name、table  name、column  name - -

MSSQL：

master..sysdatabases 、sysobjects.name、syscolumns.name

ORACLE：

SYS.ALL  TABLES 、TABLE  NAME 、SYS.ALL  TAB  COLUMNS - -

2.请求与响应分析

可以根据流量的请求与响应进行分析，从而确认拼接的 SQL 语句有没有被执行。因为 SQL 注入方式可分为 UNION 查询、报错注入、布尔盲注、时间盲注等方式。非盲注的注入方式会在响应包中存在直接的输出反馈，因此根据请求和响应分析可以比较直观明显的判断攻击是否成功。而对于盲注的注入方式，虽然不会在响应中直接输出结果，但是可以结合流量重放的方式去辅助判断。

如下图SQL 注入流量中，分析 SQL 注入语句，提取关键的语句部分简化为：

SELECT schema\_name FROM INFORMATION\_SCHEMA.SCHEMATA ，该语句作用为获取数据库名，使用了 UNION 查询的注入方式，该方式是存在回显的。因此可以直接通过分析响应数据包来判断是否为成功的注入。语句中还使用了 CONCAT 函数对数据库名拼接了一个随机字符串，随机字符串的十六进制为 71706a7171 ，解码之后为 qpjqq 。在响应包中搜索该字符串发现有匹配结果，并且该字符串后面就是 MySQL 的数据库名，可知该SQL 注入 攻击是成功的。

![](image/image_15_4Br0dxk7ar.png)

那么对于无回显的SQL 注入攻击如何判断呢。以下图中的布尔盲注为例，分析 SQL 注 入语句的意图，发现为猜解第一个数据库名的第七位字符的 ASCII 码是否大于 64。

![](image/image_16_x0H7rnBwFG.png)

可以简单修改该流量然后重放的方式判断攻击是否成功。如下，我们将猜解 ASCII码是 否为大于 64 直接改为是否大于 0 ，该语句为恒成立语句，响应数据包存在查询结果。

![](image/image_17__90dBYm5To.png)

1. 任意文件上传漏洞

漏洞概述：

任意文件上传漏洞是由于应用上传功能没有对用户上传的文件做校验或者校验不严格导致攻击者可上传任意文件，攻击者可通过任意文件上传漏洞上传 WebShell对服务器进行控制。

研判分析：

对于文件上传攻击的研判，可以先从请求和响应入手。在请求数据包中着重关注上传数 据包的文件名以及文件内容。如果文件后缀名为.php 、.php3 、.php5 、pht 、phtml、asp、asa、 cer、aspx 、asmx、jsp、jspx 、war 等，则需要特别关注。进一步需要再结合文件内容判断是 否为恶意的文件上传请求。对于文件内容是否为恶意，可以查看是否包含一些敏感的函数，常见的如命令执行、代码执行函数等，如果对于文件内容是否恶意识别存在困难，可以和常 见的 WebShell 进行特征匹配，可通过搜索引擎搜索关键代码行或者将文件内容保存在本地， 再借助 D 盾\_Web 查杀等工具进行扫描等方式确定文件是否为恶意。

![](image/image_22_1ME3wGiAro.png)

在下图文件上传数据包中，上传的文件后缀为 php ，文件内容包含 eval 、call\_user\_func 等代码执行函数，该文件为冰蝎 WebShell ，可判定上传的行为为恶意攻击行为。

对于判定是否成功，可以根据响应的数据包内容、访问上传文件的 URL 、登录服务器查 看文件是否上传成功等方式确认是否攻击成功。

1. 命令执行漏洞

漏洞概述：

命令执行漏洞，也叫命令注入漏洞，是指由于应用对用户提交的数据过滤不严格，导致攻击者可通过命令拼接构造特殊命令字符串的方法，将恶意的命令传入到可执行操作系统命令的方法中，从而造成任意命令执行。

在命令执行漏洞的利用中，比较常见的利用方法有：

1\) 利用命令执行漏洞写入 WebShell

2\) 在服务器为 Linux 系统的情况下，利用命令执行漏洞执行反弹命令，获取服务器的 Shell

3\) 在服务器通外网的情况下，执行远程下载命令，下载恶意的脚本或者木马等文件， 运行后控制服务器。

1.存在回显的情况

1. Shiro 反序列化漏洞

漏洞概述：

Shiro≤1.2.4 版本默认使用 CookieRememberMeManager ，当获取用户请求时，关键处理过程如下：

· 获取 Cookie 中 rememberMe 的值

· 对 rememberMe 进行 Base64 解码

· 使用AES 进行解密

· 对解密的值进行反序列化

由于 AES 加密的 Key 为硬编码的 Key，因此攻击者可通过使用默认的 Key 对恶意构造的 序列化数据进行加密，当 CookieRememberMeManager 对恶意的 rememberMe 进行以上过 程处理时，最终会对恶意数据进行反序列化，从而导致反序列化漏洞。另外，需要注意的是 不仅仅使用默认 Key 会存在反序列化漏洞，只要 Key 能被攻击者通过其他方式获取到，例如 使用了互联网上已公开的常见 100 个 Key ，同样会存在反序列化漏洞。

研判分析：

1.是否使用了 Shiro 框架

对于 Shiro 反序列化漏洞攻击的研判，首先可以判断应用是否使用了 Shiro 框架。判断方 法非常简单，可以通过分析攻击数据包判断或者进行简单测试判断。

攻击者在进行 Shiro 反序列化漏洞攻击时，会在请求数据包头部的 Cookie 添加  rememberMe=xxxx 字符串，如果响应数据包的头部返回 Set-Cookie: rememberMe=deleteMe 的话则应用使用了 Shiro 框架。需要注意的是如果攻击者使用构造继承 PrincipalCollection 的 空对象探测 Shiro 的 Key 时，如果 Key 正确的情况下响应包头部则不会返回 Set-Cookie: rememberMe=deleteMe ，因此在完整的 Shiro 反序列化漏洞攻击中可能会出现几个数据包不 会返回 Set-Cookie: rememberMe=deleteMe 的。

![](image/image_33_fTc3m-3PJS.png)

或者使用 burpsuite 进行测试，请求目标 URL 并抓取数据包，并且在在请求数据包的头 部添加rememberMe=xxxx 字符串，数据包的头部返回 Set-Cookie: rememberMe=deleteMe 则应用使用了 Shiro 框架。

![](image/image_34_AWf36IVIZG.png)

2.是否为误报

由于 Shiro 反序列化攻击流量为加密的流量，很多安全产品没有尝试先对加密的 rememberMe 先进行解密再检测是否有恶意特征，而是直接匹配到 Cookie 中存在一定长度的 rememberMe 就会触发告警。在使用 Shiro 框架的应用中，正常的用户登录过程中，也会出现 Cookie 中包含 rememberMe ，因此会造成一些误报。如下图，为一个正常的用户登录数据包：

![](image/image_35_Ucwfyhuw0i.png)

这种情况下可使用在线工具对加密的 rememberMe 进行解密，再判断是否存在恶意流量。 [https://simolin.cn/tools/shiro/](https://simolin.cn/tools/shiro/ "https://simolin.cn/tools/shiro/")

如下图，解密出来的内容可以看到是和用户身份相关的，没有恶意的特征，对于这种流量则是正常的。但是需要注意的是，如果正常业务流量中 rememberMe 能被解密，无论有没有遭受到攻击，Shiro 反序列化漏洞是确实存在的，下图流量中就可以知道 Shiro 加密使用了 默认的 Key。

![](image/image_36_7o8eavT2-_.png)

再如以下流量，解密出来发现存在攻击特征，ysoserial 是 JAVA 反序列化漏洞生成  payload 的工具，org.apache.commons.beanutils.BeanComparator 表明使用的 payload 为 CommonsBeanutils1 ，whoami 表示利用漏洞执行了 whoami 命令。

![](image/image_37_gtdi_JNdXE.png)

3.Shiro 版本是否为漏洞影响版本

通过登录服务器，查看应用的依赖的 Shiro 版本是否≤1.2.4 。需要注意的是并不是说Shiro>1.2.4 版本就不存在反序列化漏洞，如果开发自定义了 Shiro 的密钥为常见的 Key ，或者攻击者通过文件读取等方式获取了 Shiro 的密钥，同样会存在反序列化漏洞的风险。

![](image/image_38_K1XMHbnl3G.png)

也可以使用测试的方法辅助判断，工具链接：<https://github.com/wyzxxz/shiro_rce_tool，> 使用 shiro\_rce\_tool 工具测试是否可获取 Key ，如下图可知 Key 为默认的  kPH+bIxk5D2deZiIxcaaaA==

![](image/image_39_dpU24PXzbG.png)

4.攻击是否成功

Shiro 反序列化漏洞的利用默认是无回显的，也就是说攻击者执行恶意命令等不会在响应的数据包中显示执行命令的结果，但是攻击者可以通过构造回显的方式使得该漏洞支持回显。 对于判断攻击是否成功可以分为存在回显和没有回显两种情况进行判断。

冰蝎 (Behinder)

工具描述：

冰蝎是一款当下流行的 WebShell 客户端，因其实现了动态流量加密，使得基于流量检测 的安全产品难以准确检测出其恶意的通信流量，因此广受攻击者喜爱。

冰蝎链接：<https://github.com/rebeyond/Behinder>

研判分析：

基于流量检测的安全产品对于冰蝎的检测主要基于两个方面的流量，一个是利用文件上传漏洞上传冰蝎 WebShell 时的流量，另外一个是冰蝎服务端与客户端进行通信时的流量。

1.上传流量的检测

攻击方在利用文件上传漏洞上传冰蝎 WebShell 时，安全产品在检测到危险的文件后缀， 以及对上传的文件内容进行检测时，可以比较准确的识别出冰蝎 WebShell。

如下图出现冰蝎加密 PHP WebShell 文件上传的告警事件。

![](image/image_112_nSJrfEzrsK.png)

![](image/image_113_t3ekv3o_GZ.png)

由于响应包使用了gzip 压缩，因此在上图中看到的响应包是乱码的，可以下载数据包进行分析。

![](image/image_114__ldqjiPM5I.png)

找到对应的 HHTP 请求，右键 Follow->HTTP Stream 即可查看到 gzip 解压的响应内容。 如下图，从响应的信息可判断冰蝎 WebShell 已经被成功上传

![](image/image_115_nYSfv2Ahj6.png)

也可访问冰蝎 WebShell ，或者登录服务器排查文件的方式进一步确认冰蝎 WebShell 是 否被上传成功。如下访问发现文件存在，确认 WebShell 成功上传到了服务器。

需要注意的是除了文件上传漏洞之外，其他的漏洞比如 JAVA 反序列化漏洞或者命令执 行漏洞利用难以通过上传流量对冰蝎 WebShell 进行检测，因为这些漏洞的利用并不是通过直 接文件上传的方式写入 WebShell。

2.通信流量的检测

安全产品根据冰蝎通信的流量特征进行检测，但是由于无法对通信流量进行解密，因此 根据通信流量无法实现对冰蝎进行比较精准的检测，如检测到请求或者响应的符合加密特征 的流量会触发告警，而正常的业务通信流量也可能会匹配到这样的特征，因此会带来很多误 报的问题。在基于通信流量检测触发的告警事件中，比较重要的就是能够分析并识别误报情

况。

无文件 WebShell

攻击描述：

无文件 WebShell 不是指具体的一款 WebShell攻击工具，而是指一种攻击技术。无文件 WebShell 攻击技术允许往应用中注入 WebShell，而无需 WebShell 文件落地，以此来躲避安 全设备及安全人员的应急排查、查杀等。无文件 WebShell 一般比较广泛的用于 JAVA 应用的 漏洞攻击中，尤其是通过 JAVA 反序列化漏洞攻击注入无文件 WebShell 尤为常见。比较常见 的无文件 WebShell 实现方式有：

1\. 基于 Servlet 规范，通过动态注册 Servlet 、Filter 、Listener 等实现无文件 WebShell

2\. 基于特定框架，如 Spring 框架下动态注册 Controller 等。

3\. 基于 JAVA Agent ，如 memShell

研判分析：

由于无文件 WebShell 可以通过动态注册 Servlet 或者 Filter 等方式实现，因此攻击者可 以通过自定义任意的 URL 匹配模式来决定哪些 URL 会通过注册恶意的 Servlet 或者 Filter 来 处理。也就是说无文件 WebShell 的 URL 最后部分不再局限于任何后缀，也可以是无后缀。

# 高频命令研判

攻击者在进行远程命令执行/反序列化/MSSQL 注入等漏洞的利用时，通常会执行远程下 载/加载命令，下载 C2 等恶意文件或者脚本到受害主机执行，从而控制主机。在不同的操作 系统中使用不同的命令执行远程下载/加载命令时会产生特定的流量，可通过流量中的User-Agent 识别不同的命令执行流量，再结合下载的文件类型等进行流量检索， 以识别流量 中可能存在的恶意命令执行流量。

&#x20;具体含义参考每个命令部分 ：&#x20;

(user\_agent:*BITS* AND r\_body:MZ *) (user\_agent:Mozilla*compatibleTriden*NET* AND r\_body:*scriptlet*) (user\_agent:*Mozilla*compatible*Triden*NET\* AND (r\_body:\* Wscript.Shell\* OR  r\_body:*VBScript* OR r\_body:\* CreateObject *)) OR ((user\_agent:Wget* OR user\_agent: curl) AND (r\_body:*ELF* OR r\_body:*socket* OR r\_body:*exec* OR  	(r\_ body:*dev* AND r\_body:*tcp*)) OR (user\_agent:*Java* AND r\_body:*java*) OR (user\_agent:*Python* AND (r\_body:*exec* OR r\_body:*socket*)) OR  (protocol\_type:"http" AND (user\_agent:*WindowsPowerShell* OR user\_agent:"") AND (r\_body:*New-Object* OR r\_body:*Decompress* OR r\_body:*Assembly* OR r\_body:*nishang* OR r\_body:*PowerSploit* OR r\_body:MZ \*))
